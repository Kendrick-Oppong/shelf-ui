Every interactive Shelf UI component ships with two implementations: one built on **Radix UI**, one built on **Base UI**. The component API is identical — only the internal primitive layer differs. You choose once, at project init, and never revisit it.

## What are primitives?

Primitives are unstyled, accessible, headless components that handle behavior — keyboard navigation, focus management, ARIA attributes, screen reader announcements — without imposing any visual opinions. Shelf UI components are built on top of them.

We support two primitive libraries:

| | Radix UI | Base UI |
| --- | --- | --- |
| **Maintained by** | WorkOS | MUI team |
| **Used in** | shadcn/ui, Vercel, Linear | MUI ecosystem |
| **Maturity** | Production-proven, ~5 years old | Newer, rapidly growing |
| **Bundle size** | Per-package (tree-shakeable) | Per-package (tree-shakeable) |
| **React compatibility** | React 16+ | React 18+ |
| **WAI-ARIA** | Full compliance | Full compliance |
| **Animation** | Via CSS or Framer Motion | Via CSS or Framer Motion |

## How to choose

<Callout type="info">
  This is exactly how shadcn/ui handles dual primitives. The pattern is proven — pick one and move on.
</Callout>

<Tabs items={["New project", "Existing shadcn/ui project", "Existing MUI project"]}>
  <Tab value="New project">
    **Use Radix UI.**

    It has the largest ecosystem, the most third-party examples, and the deepest shadcn/ui integration. If you're starting fresh, Radix is the safer default.

    ```bash title="terminal"
    # When prompted by the Shelf UI CLI:
    ? Which primitive layer? › Radix UI
    ```
  </Tab>
  <Tab value="Existing shadcn/ui project">
    **Use Radix UI.**

    Your project already has Radix UI installed as a peer dependency. Shelf UI's Radix variant will slot in with zero additional dependencies.
  </Tab>
  <Tab value="Existing MUI project">
    **Use Base UI.**

    If you're already using MUI's Base UI primitives elsewhere in your app, use the Base UI variant to stay consistent and avoid shipping two primitive libraries.
  </Tab>
</Tabs>

## Switching later

Switching is zero-cost. The import path is the same for both variants:

```tsx title="Your component — unchanged regardless of primitive"
import { Dropzone } from "@/components/shelf-ui/dropzone";

// Props, events, and behavior are identical
<Dropzone onDrop={handleDrop} accept={["image/*"]} maxSize={10_000_000} />
```

The only difference is the file that gets copied into your project. To switch, delete the component file and re-add it with the other primitive:

```bash title="terminal"
# Remove the Radix version
rm components/shelf-ui/dropzone.tsx

# Re-add with Base UI
npx shadcn@latest add @shelf-ui/dropzone --primitive base
```

## Headless hooks

Every interactive component ships with its underlying hook. Use the hook directly if you want to build a completely custom UI:

```tsx title="Using the hook without the pre-built component"
import { useFileUpload } from "@/hooks/use-file-upload";

export function CustomDropzone() {
  const { files, isDragging, open, getRootProps, getInputProps } = useFileUpload({
    accept: ["image/*"],
    maxSize: 5_000_000,
    onDrop: (accepted) => console.log(accepted),
  });

  return (
    <div
      {...getRootProps()}
      data-dragging={isDragging}
      className="my-custom-zone"
    >
      <input {...getInputProps()} />
      {isDragging ? "Drop files here" : "Drag files or click to upload"}
    </div>
  );
}
```

The hook is identical regardless of which primitive variant you installed.

## FAQ

<Accordions type="single">
  <Accordion title="Do I need to install Radix UI or Base UI globally?">
    No. The shadcn CLI handles peer dependency installation when you add each component. You only install what each component actually needs.
  </Accordion>
  <Accordion title="Can I mix Radix and Base UI components in the same project?">
    Technically yes, but it's not recommended — you'd ship two primitive libraries. Pick one and stick with it across all Shelf UI components.
  </Accordion>
  <Accordion title="Which primitive is more performant?">
    In practice, both are negligible in isolation. The performance difference is not a valid selection criterion. Choose based on ecosystem fit.
  </Accordion>
  <Accordion title="What happens if a component doesn't need a primitive?">
    Components like `FileCard`, `FileIcon`, and `StorageQuota` are pure display components. They don't use Radix or Base UI at all — there's no choice to make for these.
  </Accordion>
</Accordions>
